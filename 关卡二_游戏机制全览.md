## 二、阶段状态机（Phase State Machine）

关卡采用严格的线性阶段推进模型，共 **7 个阶段**。阶段决定了当前可用的行动卡、场景交互和高亮提示。

```
S0_SCENE_ASSESSMENT → S1_RESPONSE_CHECK → S2_CALL_AND_AED → S3_CPR_MAINTAIN → S4_AED_USE → S5_WAIT_AMBULANCE → S6_RESULT
```

| 阶段 ID | 阶段名称 | 进入条件 | 核心行为 |
|---------|----------|----------|----------|
| `S0_SCENE_ASSESSMENT` | 现场安全判断 | 游戏开始 | 点击高亮 CORE 区域确认安全 |
| `S1_RESPONSE_CHECK` | 检查反应 | 完成安全判断 | 移动到老人相邻格，轻拍双肩呼唤 |
| `S2_CALL_AND_AED` | 呼救与 AED | 确认无反应 | 依次指派路人 A 打 120、路人 B 取 AED、判断呼吸 |
| `S3_CPR_MAINTAIN` | CPR 维持 | 确认无正常呼吸 | 节奏 QTE 按压，疏散通道，等待 AED |
| `S4_AED_USE` | AED 步骤 | 路人 B 送达 AED | 四步操作 AED（打开→分析→离身→除颤） |
| `S5_WAIT_AMBULANCE` | 继续 CPR | AED 除颤完成 | 持续 CPR + 周期性 AED 再分析，等待救护车 |
| `S6_RESULT` | 结算复盘 | 救护车到达 或 失败条件触发 | 展示评分、行为总结、知识卡 |

### 2.1 阶段推进保护

每个阶段的入口都有**前置条件校验**，防止跳步。例如：

- 未完成安全判断时点击非 CORE 区域 → 报错「请先判断现场安全」
- 未打 120 就指派 AED → 报错「先让路人 A 拨打 120」
- 未在老人相邻格就检查呼吸 → 报错「需要在老人相邻格判断呼吸」
- CPR 阶段点击其他格子 → 报错「CPR 阶段不能离开老人」

## 三、行动力系统 (AP System)

### 3.1 基本规则

| 属性 | 值 |
|------|-----|
| 每回合 AP 上限 | 4 点 |
| AP 恢复时机 | AP 耗尽时自动推进回合（`advanceProcedureTurn`），重置为 4 |
| 移动消耗 | 1 AP / 格（仅相邻格可移动） |
| 决策卡消耗 | 大多数正确选项消耗 1 AP |
| 错误选项消耗 | 错误行动同样消耗 1 AP（通过 `applyWrongAction` 扣减） |

### 3.2 回合推进机制 (`advanceProcedureTurn`)

当 AP 耗尽且 CPR 尚未开始时触发：

- `round` 计数器 +1
- AP 重置为 `MAX_AP`（4 点）
- 如果已完成反应检查但尚未拨打 120，生命稳定度额外 -5（惩罚拖延呼救）

### 3.3 CPR 阶段的 AP 变化

进入 CPR 阶段后，AP 系统实际上**不再消耗**。CPR 按压和疏散通道使用**指挥点**（Command Point）而非 AP。每次 CPR 按压后 `commandPoint` 重置为 1。

## 四、生命稳定度系统 (Stability)

### 4.1 数值范围与视觉反馈

- 范围：0 ~ 100，使用 `clamp` 严格限制
- 顶部 HUD 显示数值 + 进度条
- 进度条颜色分段：
  - **绿色** (`0x49bf6b`)：stability > 65
  - **黄色** (`0xffd764`)：35 < stability ≤ 65
  - **红色** (`0xe34f44`)：stability ≤ 35
- 进度条宽度 = `barWidth × (stability / 100)`

### 4.2 稳定度变化汇总表

| 触发事件 | 变化量 | 备注 |
|----------|--------|------|
| CPR 命中（绿色区域） | -1 | 模拟按压消耗，但维持循环 |
| CPR 失误（非绿色区域） | -6 | 节奏偏差导致灌注不足 |
| 错误行动（严重） | -8 ~ -20 | 取决于选项的 `stabilityLoss` 参数 |
| 错误行动（非严重） | -5 ~ -10 | `severe = false` |
| 拖延呼救（回合推进惩罚） | -5 | 已检查反应但未打 120 |
| 人工呼吸正确 | +3 | 30:2 循环中的恢复 |
| AED 除颤成功 | +15 | 电击后瞬间恢复 |
| AED 完成（`finishAedUse`） | +4 | 进入 S5 阶段时的奖励 |

### 4.3 失败判定

`stability ≤ 0` 时立刻触发失败，显示：「救援流程被耽误太久了。下次先让 120 在路上，再尽快开始 CPR。」

## 五、决策卡系统 (Decision Card System)

### 5.1 机制概述

每个阶段通过**决策弹窗**（`showDecisionCard`）呈现选择题形式的交互。弹窗包含标题、描述正文和 2~4 个选项。选项在 UI 上统一为"中性选择题"样式（A/B/C/D），**不提前透露正确答案**。

### 5.2 选项分类

| 类型 | 标记属性 | 行为 |
|------|----------|------|
| 正确选项 | `recommended: true` | 消耗 1 AP，推进阶段，加分 |
| 错误选项（严重） | `danger: true`（默认 `severe = true`） | 扣 AP + 扣稳定度 + 严重错误计数 +1 |
| 错误选项（非严重） | `danger: true`, `severe = false` | 扣 AP + 扣稳定度，不计严重错误 |
| 不完整选项 | 无特殊标记 | 消耗 1 AP，不推进阶段，给出提示 |

## 六、CPR 节奏 QTE 系统

### 6.1 QTE 核心设计

CPR 节奏 QTE 是本关卡的**核心操作机制**，模拟真实心肺复苏的按压频率（100~120 次/分钟）。

**视觉构成：**
- 底部行动面板左侧有一条 **308px 宽的节奏条**
- 节奏条上有 **两段绿色命中区域**
- 一个红色三角标记（`qteMarker`）在节奏条上**持续左右摆动**

### 6.2 指针摆动公式

```javascript
getQteX(time) {
  const t = (Math.sin(time / 950) + 1) / 2;   // 正弦波归一化到 [0, 1]
  return qteBar.x + t * qteBar.width;           // 映射到节奏条像素范围
}
```

- 使用 `sin(time / 950)` 控制摆动，周期约 `950 × 2π ≈ 5969ms ≈ 6 秒`
- 模拟频率约为 **110 次/分**（标准 CPR 节奏）
- 指针在节奏条上平滑往复运动

### 6.3 命中判定规则

节奏条总宽 308px，起点偏移 `barX`：

| 区域 | 像素范围（相对 barX） | 对应含义 |
|------|----------------------|----------|
| 过快区 | 0 ~ 113 | 按压频率 > 130 次/分 |
| **绿色命中区 1** | **114 ~ 180** | 频率约 100~120 次/分 |
| 间隙盲区 | 181 ~ 219 | 节奏不稳 |
| **绿色命中区 2** | **220 ~ 262** | 频率约 100~120 次/分（第二窗口） |
| 过慢区 | 263 ~ 308 | 按压频率 < 90 次/分 |

判定逻辑：指针当前 x 坐标落在**绿色区域 1 或 2**内即为命中（`isQteHit`）。

### 6.4 按压反馈

| 结果 | 稳定度变化 | 反馈文本 | 音效 |
|------|-----------|----------|------|
| 命中 | -1 | 「按压有效 (~110次/分)」绿色飘字 | `cpr_hit` |
| 失误-过快区 | -6 | 「按压过快!」红色飘字 + 科普说明 | `error` |
| 失误-过慢区 | -6 | 「按压过慢!」红色飘字 + 科普说明 | `error` |
| 失误-间隙区 | -6 | 「节奏不稳!」红色飘字 + 科普说明 | `error` |

飘字效果：文字从角色上方弹出，900ms 内向上漂浮并淡出（alpha 从 1 渐变到 0）。

### 6.5 QTE 活跃条件

指针仅在以下条件全部满足时可见并移动：

- 当前阶段为 `S3_CPR_MAINTAIN` 或 `S5_WAIT_AMBULANCE`
- AED 未处于分析状态（`!aedAnalyzing`）
- 无模态弹窗打开（`!modalOpen`）

### 6.6 CPR 节奏条视觉构成

| 元素 | 位置 (相对 barX) | 宽度 | 颜色 |
|------|-----------------|------|------|
| 底层条 | 0 | 308 | `0xfcf5e8`（浅米色） |
| 绿色区 1 | 114 | 66 | `0x2fbf7a`（抹茶绿） |
| 绿色区 2 | 220 | 42 | `0x2fbf7a`（抹茶绿） |
| 红色指针 | 动态 | 三角形 22×20 | `0xe34f44`（红色） |

底部标签根据命中率动态更新：
- 命中率 ≥ 70%：「频率极佳 (~110次/分)」
- 命中率 45%~69%：「频率偏离 (~90或~130次/分)」
- 命中率 < 45%：「偏离标准频率! (请跟随指针节奏)」

## 七、30:2 人工呼吸循环机制

### 7.1 触发条件

每累计 **6 次有效 CPR 命中**（`cprCount >= 6`），触发人工呼吸决策卡。这是对真实急救中 **30 次按压 : 2 次吹气** 比例的游戏化简化。

### 7.2 执行流程

1. `cprCount` 归零
2. 弹出「人工呼吸」决策卡（`showRescueBreathsCard`）
3. 选择正确：稳定度 +3，路人 B 前进 3 格，刷新场景
4. 选择错误：稳定度 -5，重新弹出该卡（`persist: true`），直到选对

### 7.3 参数

| 参数 | 值 |
|------|-----|
| 触发阈值 | 6 次有效按压 |
| 正确恢复量 | +3 稳定度 |
| 错误惩罚量 | -5 稳定度 |
| 错误是否严重 | 否 (`severe = false`) |

## 八、AED 周期再分析机制（S5 阶段）

### 8.1 触发条件

在 `S5_WAIT_AMBULANCE` 阶段（AED 已使用后的持续 CPR 阶段），每隔 **2 次 CPR 按压**触发一次 AED 心律再分析。

### 8.2 执行流程

1. `aedAnalysisTimer` 每次按压 +1
2. 达到 2 时重置计时器，设置 `aedAnalyzing = true`
3. 弹出「AED 心律分析警告」卡
4. 正确选择（全体避让）→ 弹出「分析结果：建议电击」卡 → 正确执行后稳定度 +15
5. 错误选择（继续按压或触碰患者）→ 严厉惩罚（-15 或 -20 稳定度）

### 8.3 安全联锁

在 `aedAnalyzing === true` 期间：

- CPR 按压操作被拦截，弹出错误提示
- QTE 指针隐藏
- 必须通过决策卡确认「全体避让」才能解除

### 8.4 参数

| 参数 | 值 |
|------|-----|
| 再分析间隔 | 每 2 次 CPR 按压 |
| 正确避让奖励 | 稳定度 +15（电击效果） |
| 触碰患者惩罚（分析中） | -15 稳定度 + 严重错误 |
| 放电时触碰惩罚 | -20 稳定度 + 严重错误 |

## 九、AED 操作机制（S4 阶段）

### 9.1 四步流程卡

AED 使用通过步骤递进的决策卡实现，每步只能顺序执行，**不允许跳步**。

| 步骤 | aedStep 值 | 操作内容 | 得分 |
|------|-----------|----------|------|
| 第 1 步 | 0 | 打开 AED | 基础 6 + step×3 = 9 |
| 第 2 步 | 1 | 等待设备分析 | 6 + 2×3 = 12 |
| 第 3 步 | 2 | 确认所有人离开身体 | 6 + 3×3 = 15 |
| 第 4 步 | 3 | 执行除颤并继续 CPR | 满分 20 |

### 9.2 离身确认保护

- 在第 3 步完成前（`allClear === false`），「直接除颤」选项被标记为 `danger`
- 点击未离身的直接除颤 → 报错「除颤前必须确认所有人离开身体」，稳定度 -8
- 只有 `allClear === true` 后，除颤才能执行

### 9.3 每步附带的持续选项

每个 AED 步骤卡片底部固定提供两个额外选项：

- **「直接除颤」**：在 `allClear === false` 时标记为 `danger`，点击会被拦截并扣 8 稳定度；仅在完成离身确认后可正常执行
- **「暂时返回 CPR」**：关闭 AED 卡片，继续按压。花花会提醒「AED 步骤要尽快完成，但不要跳过离身确认」

这意味着玩家在任何步骤都可以选择回退或尝试跳步（被系统阻止），而非强制线性操作。

### 9.4 完成后效果

- 阶段切换至 `S5_WAIT_AMBULANCE`
- `aedUsedCorrectly = true`
- 稳定度 +4
- AED 评分锁定为 20 分
- 后续 CPR 进入「AED 周期再分析」模式

## 十、路人 B 寻路系统（NPC Pathfinding）

### 10.1 路线定义

路人 B 的移动路径由两组**固定坐标序列**定义：

**去程（`AED_ROUTE_TO`）**：从 (2,5) 到 (8,4)，共 8 个节点

```
(2,5) → (3,5) → (4,5) → (4,4) → (5,4) → (6,4) → (7,4) → (8,4)
```

**回程（`AED_ROUTE_BACK`）**：从 (8,4) 到 (2,4)，共 7 个节点

```
(8,4) → (7,4) → (6,4) → (5,4) → (4,4) → (3,4) → (2,4)
```

### 10.2 移动规则

| 参数 | 值 |
|------|-----|
| 每次最大步数 | 3 格（`advanceBystanderB(3)`） |
| 移动触发时机 | 每次 CPR 按压后 或 正确决策后 |
| 状态机 | `idle` → `toAED` → `withAED` → `handedOff` |

### 10.3 移动算法

```
while (steps < maxSteps && routeIndex < route.length - 1):
  查看下一个节点
  if 下一个节点有围观群众:
    标记该群众为 blocking
    aedDelay += 1
    停止移动，提示玩家疏散
    break
  else:
    routeIndex += 1
    更新路人 B 坐标
    steps += 1
```

### 10.4 状态转换

| 当前状态 | 到达终点后 | 效果 |
|----------|-----------|------|
| `toAED` | → `withAED` | routeIndex 归零，切换回程路线，播放 pickup 音效 |
| `withAED` | → `handedOff` | `aedDelivered = true`，阶段切换到 S4，播放 pickup 音效 |

### 10.5 方向动画

路人 B 根据下一步的移动方向自动播放对应奔跑动画：

- `dx > 0` → `l2_b_run_right`（8fps）
- `dx < 0` → `l2_b_run_left`
- `dy < 0` → `l2_b_run_up`
- `dy > 0` → `l2_b_run_down`

## 十一、围观群众系统 (Crowd System)

### 11.1 群众初始配置

| ID | 坐标 | 贴图 | 气泡台词 | 阻挡标记 |
|----|------|------|----------|----------|
| c1 | (3, 2) | crowd_a | 「先别乱动」 | false |
| c2 | (4, 3) | crowd_b | 「给他喝点水？」 | false |
| c3 | (6, 4) | crowd_c | 「我看看」 | **true**（阻挡 AED 路径） |
| c4 | (7, 3) | crowd_d | 「AED 在哪边？」 | false |

### 11.2 阻挡机制

- 群众 c3 初始位于 AED 路线上 `(6, 4)`
- 当路人 B 移动到该节点时，检测到群众占格
- 群众被标记为 `blocking = true`，气泡变为「请让出通道」
- 路人 B **停止移动**，`aedDelay` +1
- 场景出现橙色高亮标记提示玩家疏散

### 11.3 疏散机制

**触发条件：**
- CPR 阶段（S3 或 S5）
- 拥有 1 指挥点（`commandPoint > 0`）
- 存在阻挡 AED 路径的群众

**执行流程 (`clearAedPath`)：**
1. 消耗 1 指挥点（`commandPoint = 0`）
2. 找到第一个阻挡路径的群众（`isCrowdOnAedPath`）
3. 将其 `blocking` 设为 false，气泡改为「好，我们后退」
4. 寻找合法移动目标格（`findCrowdMoveCell`）
5. 移动群众到新位置
6. 场景分 +4

### 11.4 群众移动目标选择 (`findCrowdMoveCell`)

按优先级依次检查以下 5 个候选方向：

```
(x, y+1)  → 正下方
(x+1, y+1) → 右下
(x-1, y+1) → 左下
(x+1, y)   → 正右
(x-1, y)   → 正左
```

合法条件：在网格范围内，且目标格不被老人、玩家、AED 柜、路人 A/B、其他群众占据，也不在 AED 路线上。

所有候选均不合法时，回退到默认位置 (7, 5)。

### 11.5 场景点击疏散

CPR 阶段，玩家可以直接点击阻挡群众的格子来触发疏散（无需通过行动卡），系统自动调用 `clearAedPath()`。

## 十三、指挥点系统 (Command Point)

### 13.1 规则

| 属性 | 值 |
|------|-----|
| 获取时机 | 每次 CPR 按压后重置为 1 |
| 初始获取 | 进入 S3 时初始化为 1 |
| 上限 | 1 点 |
| 消耗场景 | 疏散围观群众（`clearAedPath`） |

### 13.2 设计意图

指挥点机制确保玩家**不能同时做 CPR 和疏散**——每次按压后获得 1 点指挥点，用于疏散；下次按压时指挥点不额外增加（重置为 1 而非累加）。这迫使玩家在按压和通道管理之间做出节奏选择。

## 十四、评分系统 (Scoring System)

### 14.1 分值构成

| 评分维度 | 变量名 | 获取方式 | 最大可得分 |
|----------|--------|----------|-----------|
| 流程正确性 | `score.procedure` | 安全判断 +8, 反应检查 +8, 进入 CPR +14 | 30 |
| 协作指挥 | `score.cooperation` | 指派 120 +10, 指派 AED +8 | 18 |
| CPR 质量 | `score.cpr` (计算) | `round(命中率 × 14)` + 按压≥5次额外 +4 | 18 |
| AED 操作 | `score.aed` | 正确使用 20 / 仅送达 8 | 20 |
| 场景管理 | `score.scene` | 疏散通道 +4（含基础分）, 通道清理 +3 | 7 |
| 成功奖励 | — | 成功救援 | 8 |

### 14.2 扣分项

| 扣分类型 | 扣分公式 |
|----------|----------|
| 错误行动 | `wrongActions × 4` |
| 严重错误 | `severeErrors × 10` |
| AED 延迟 | `min(aedDelay, 3) × 2` |

### 14.3 最终得分计算

```javascript
score = procedure + cooperation + round(cprRate × 14)
      + (cprAttempts >= 5 ? 4 : 0)
      + (aedUsedCorrectly ? 20 : aedDelivered ? 8 : 0)
      + scene + (pathCleared ? 3 : 0)
      + (success ? 8 : 0)
      - wrongActions × 4
      - severeErrors × 10
      - min(aedDelay, 3) × 2

finalScore = clamp(score, 0, 100)
```

### 14.4 评级标准

| 评级 | 条件 |
|------|------|
| S 级 | 总分 ≥ 90 **且** AED 正确使用 **且** 严重错误 = 0 |
| A 级 | 总分 ≥ 75 |
| B 级 | 总分 ≥ 60 |
| 需要复盘 | 总分 < 60 或救援失败 |

## 十六、错误行动与惩罚系统

### 16.1 `applyWrongAction` 函数逻辑

```
applyWrongAction(message, stabilityLoss, severe = true):
  1. 如果 AP > 0 且 CPR 未开始：AP - 1
  2. 稳定度 -= stabilityLoss
  3. wrongActions += 1
  4. 如果 severe：severeErrors += 1
  5. 播放 error 音效 (0.72 音量)
  6. 花花显示纠正提示
  7. 如果 AP <= 0 且 CPR 未开始：推进回合
  8. 检查失败条件
  9. 刷新场景
```

### 16.2 `feedbackError` 函数（轻量反馈）

用于操作受限时的提示（如点击被阻挡的格子、CPR 阶段尝试移动、超出网格范围等），**不消耗 AP**。

**但需注意**：此函数同样会令 `wrongActions += 1`，而评分公式中 `wrongActions × 4` 会扣分。这意味着反复误点被阻挡的格子也会累积扣分，玩家需要注意操作的精确性。

- `wrongActions += 1`
- 播放 error 音效 (0.52 音量)
- 镜头微震（90ms, 0.002 振幅），650ms 冷却防抖
- 花花显示纠正文案

### 16.3 全部错误选项及惩罚参数汇总

| 所在决策卡 | 错误选项 | 稳定度损失 | 是否严重 |
|-----------|----------|-----------|----------|
| 安全判断 | 立刻扶起老人 | -15 | 是 |
| 安全判断 | 给老人喝水 | -10 | 是 |
| 安全判断 | 掐人中试试看 | -8 | 否 |
| 检查反应 | 用力摇晃老人 | -8 | 是 |
| 检查反应 | 等他自己醒来 | -10 | 是 |
| 指派 120 | 请帮我拍视频 | -6 | 否 |
| 指派 120 | 请把老人扶起来 | -15 | 是 |
| 指派 AED | 我自己跑去拿 AED | -10 | 是 |
| 指派 AED | 先不用 AED | -8 | 否 |
| 判断呼吸 | 等救护车来 | -12 | 是 |
| 判断呼吸 | 给老人喝水 | -10 | 是 |
| 判断呼吸 | 让围观者扶坐 | -15 | 是 |
| 人工呼吸 | 不捏鼻子吹气 | -5 | 否 |
| 人工呼吸 | 压胸口吹气 | -5 | 否 |
| AED 分析 | 继续按压 | -15 | 是 |
| AED 电击 | 拉住老人手放电 | -20 | 是 |
| AED 步骤 | 未离身直接除颤 | -8 | 是 |
| CPR 中 | AED 分析时触碰患者 | -15 | 是 |

1. 模态窗口打开或游戏结束 → 忽略

3. CPR 阶段锁定检查
4. 安全判断完成检查
5. 相邻格距离检查
6. 阻挡原因检查
7. AP 消耗
8. 执行移动

### 21.2 人群环境音逻辑

`updateCrowdAmbience` 控制背景嘈杂音的播放：

- **播放条件**：游戏未结束 **且** (处于 S0/S1/S2 阶段 **或** 存在阻挡群众)
- **停止条件**：游戏结束 **或** `crowdForceStopped = true`（游戏开始 10 秒后强制停止）
- 10 秒自动停止设计：开场给玩家一段嘈杂氛围感后安静下来，避免长时间干扰

## 二十二、角色动画系统

### 22.1 预定义动画

| 动画 Key | 帧序列 | 帧率 | 用途 |
|----------|--------|------|------|
| `l2_player_cpr` | press, release | 4 fps | 玩家 CPR 按压动作循环 |
| `l2_a_call` | call_1, call_2 | 5 fps | 路人 A 打电话动作循环 |
| `l2_b_run_right` | run_right_1, run_right_2 | 8 fps | 路人 B 向右奔跑 |
| `l2_b_run_left` | run_left_1, run_left_2 | 8 fps | 路人 B 向左奔跑 |
| `l2_b_run_up` | run_up_1, run_up_2 | 8 fps | 路人 B 向上奔跑 |
| `l2_b_run_down` | run_down_1, run_down_2 | 8 fps | 路人 B 向下奔跑 |

所有动画 `repeat: -1`（无限循环）。

### 22.2 角色状态驱动贴图

- **玩家**：CPR 开始后使用 `player_cpr_press` + 播放动画，之前使用 `player_idle`
- **路人 A**：打电话状态使用 `bystander_a_call_2` + 播放动画，否则 `bystander_a_idle`
- **路人 B**：根据移动方向选择对应奔跑动画，静止时使用 `bystander_b_idle`
- **老人**：始终使用 `fallen_elderly` 静态图，状态通过头顶标签文字表达

### 22.3 老人状态标签

| 条件优先级 | 标签文字 | 说明 |
|-----------|----------|------|
| 1（最高） | 专业接手 / 倒地 | S6 结算阶段，成功显示「专业接手」，失败显示「倒地」 |
| 2 | AED 已介入 | `aedUsedCorrectly === true` |
| 3 | CPR 进行中 | `cprStarted === true`（一旦 CPR 开始，此标签优先于「无正常呼吸」） |
| 4 | 无正常呼吸 | `hasCheckedBreathing === true`（仅在 CPR 开始前的短暂窗口可见） |
| 5 | 无反应 | `hasCheckedResponse === true` |
| 6（最低） | 倒地 | 默认状态 |

> **注意**：`getElderStatus()` 按优先级从高到低依次检查，`cprStarted` 优先于 `hasCheckedBreathing`，因此一旦进入 CPR，「无正常呼吸」标签不再显示。

## 二十三、场景重绘与渲染管线

### 23.1 图层架构

从下到上共 7 个图层容器（Phaser Container）：

| 图层 | 用途 |
|------|------|
| `staticLayer` | 永久不变的 UI 底板、顶栏、底栏 |
| `mapLayer` | 网格地砖、路径线、引导标记、装饰 |
| `actorLayer` | 所有角色精灵、标签、气泡 |
| `effectLayer` | 光环、阴影、飘字等特效 |
| `actionLayer` | CPR 节奏条、行动卡片、物品槽 |
| `uiLayer` | 顶部 HUD、花花助手面板、功能按钮 |
| `modalLayer` | 决策弹窗（最高层级） |

### 23.2 刷新机制

`refreshScene()` 是最核心的渲染函数，每次状态变更后调用：

1. 清空 `mapLayer`、`actorLayer`、`effectLayer`、`actionLayer`
2. `drawMap()` → 重绘地砖、路径、核心区、交互提示
3. `drawActors()` → 重绘所有角色、群众、AED 设备
4. `drawHud()` → 更新所有数值显示
5. `drawActionPanel()` → 重建 CPR 条和行动卡片
6. `updateCrowdAmbience()` → 更新环境音状态

这种「全量重绘」模式简化了状态同步，确保 UI 与数据始终一致。

## 二十四、辅助与无障碍设计

### 24.1 功能按钮

右上角提供「重新开始」按钮（↻），点击调用 `scene.restart()` 重置所有状态。

### 24.2 引导系统

- 每个阶段通过 `drawInteractionHints` 在地图上叠加高亮引导标记
- 花花助手持续更新当前目标和操作提示
- 推荐行动卡片有黄色脉冲发光 + 「建议」贴纸
- CPR 节奏条底部有频率标签实时反馈命中率

### 24.3 错误防护

- 流程顺序严格锁定，每一步都有前置条件校验
- AED 除颤前强制离身确认
- CPR 阶段锁定玩家移动
- AED 分析/放电期间禁止 CPR 按压
- 所有错误操作都有明确的纠正文案和科普说明

### 24.4 科普知识卡

结算卡片底部固定包含急救知识总结：「明确指派具体的人拨打 120 和取 AED；除颤前必须确认所有人离开身体；AED 后继续 CPR 直到接手。」
