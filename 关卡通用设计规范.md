# 《红花救援队》关卡通用设计规范

## 一、文档定位

本规范用于统一《红花救援队》所有正式关卡的界面布局、视觉配色、网格表现、交互反馈、AI助手提示、评分结算和开发配置方式。

关卡可以在题材、数值和特殊机制上差异化，但必须保持一致的界面骨架、交互语言和公益表达原则。玩家进入任意关卡后，应能快速理解“看哪里、点哪里、如何判断是否做对”。

## 二、设计原则

| 原则 | 说明 |
| --- | --- |
| 公益优先 | 不让危险行为或错误急救动作成为最优解 |
| 清晰可读 | 全龄玩家可理解，按钮大、文字短、状态明确 |
| 低压挑战 | 保留紧张感，但不因现实思考时间惩罚玩家 |
| 统一骨架 | 顶部状态栏、中部主玩法区、右侧花花提示、底部行动卡保持一致 |
| 机制差异 | 关卡一可用烟雾探索，关卡二可用CPR QTE，但都继承回合、反馈、复盘规则 |
| 可开发落地 | 所有状态、颜色、按钮、图标都能配置化，不依赖一次性美术特例 |

## 三、通用屏幕布局

默认采用横版16:9布局，适配平板、PC演示和横屏移动端。

| 区域 | 位置 | 功能 |
| --- | --- | --- |
| 顶部状态栏 | 屏幕顶部，高度约8%-10% | 显示AP/回合/救助对象状态/风险或倒计时 |
| 主玩法区 | 屏幕中部偏左，占宽度约65%-72% | 显示网格地图、场景、NPC、道具、危险区域 |
| 花花提示区 | 屏幕右侧，占宽度约22%-28% | 显示AI助手头像、阶段提示、纠错说明 |
| 底部行动卡 | 屏幕底部，高度约18%-24% | 显示当前可执行动作，最多4个主选项 |
| 图例/小提示 | 左下或主玩法区角落 | 显示安全区、烟雾、火焰、障碍、道具等图例 |

### 1. 布局要求

- 主玩法区始终优先展示当前救援场景。
- 行动卡不遮挡关键NPC和目标格。
- 花花提示区不承担主要操作，避免玩家误以为AI会自动完成救援。
- QTE出现时可以占据主玩法区一部分，但不能完全切断玩家对关键现场进度的理解。

### 2. 关卡二特殊布局

关卡二进入CPR阶段后：

- 网格地图缩为辅助视图，持续显示路人B取AED进度。
- CPR QTE成为主操作区域。
- 底部行动卡切换为“指挥点”操作。
- 顶部状态栏显示生命稳定度、救护车倒计时、AED进度。

## 四、视觉配色规范

配色参考关卡一效果图：整体温暖、清晰、公益向，有紧张感但不恐怖化。危险区使用橙红，安全区使用青绿色，UI底色使用深蓝灰和暖白，避免一整屏单一色调。

| 安全绿 | `#2FBF7A` | 安全区、撤离出口、正确反馈 |

| 火焰橙 | `#F47A3D` | 火焰格、火势扩散提示 |

### 3. UI颜色

| UI元素 | 颜色规则 |
| --- | --- |

| 状态 | 颜色 |

| 危险/生命下降 | `#F47A3D` |
| 高危/即将失败 | `#E14532` |
| 未知/不可见 | `#2E343B` |
| 已完成 | `#42C7B8` |

## 五、字体与文字规范

### 1. 字体层级

| 层级 | 用途 | 建议字号 |
| --- | --- | --- |
| H1 | 关卡标题、结算标题 | 28-34px |
| H2 | 阶段标题、面板标题 | 20-24px |
| 正文 | 花花提示、行动说明 | 16-18px |
| 状态数字 | AP、回合、生命状态 | 18-22px |
| 图例/次级说明 | 小地图说明、辅助文本 | 13-15px |

### 2. 文案规则

- 行动按钮文案控制在2-8个中文字符。
- 花花提示每次不超过2行。
- 错误反馈必须说明原因，不只说“错了”。
- UI内不写长教程，复杂知识放到通关复盘和知识卡。
- 避免把医疗或消防建议写成绝对化承诺。

## 六、网格地图规范

### 1. 网格尺寸

| 项目 | 规范 |
| --- | --- |
| 常规地图 | 横向10格左右，纵向7-8格 |
| 单格尺寸 | 以屏幕宽度适配，保证NPC与道具可识别 |
| 网格线 | 低对比浅线，危险区可加强描边 |
| 坐标逻辑 | 左上为原点，关卡配置中统一使用`x,y` |

### 2. 格子状态

| 状态 | 表现 |
| --- | --- |
| 可行走 | 地面正常显示 |
| 当前可移动 | 青色或白色边框高亮 |
| 目标格 | 暖黄描边或图标提示 |
| 安全区 | 绿色底纹和出口箭头 |
| 烟雾格 | 半透明灰雾覆盖 |
| 未知格 | 深灰遮罩，保留边界 |
| 火焰格 | 橙红底和火焰动效，不可通行 |
| 障碍格 | 物件占格，显示清障图标 |
| NPC格 | 头像或角色模型占格，显示状态条 |

### 3. 交互反馈

- 鼠标/手指悬停可显示格子类型。
- 点击不可通行格时，显示原因而不是无反馈。
- 路径预览用细线或格子边框，不用大面积实色覆盖。
- 关键目标可以轻微脉冲，但同屏脉冲目标不超过2个。

## 七、状态栏规范

顶部状态栏用于表达“当前救援压力”，不同关卡可替换字段，但位置和样式统一。

### 1. 通用字段

| 字段 | 显示规则 |
| --- | --- |
| 回合 | `回合 1/12` |
| 行动资源 | `AP 4/4`或`指挥点 1/1` |
| 救助对象状态 | `生命状态 100`或`生命稳定度 100` |
| 风险/倒计时 | 关卡一显示火场风险，关卡二显示救护车倒计时 |

### 2. 关卡配置示例

| 关卡 | 状态栏字段 |
| --- | --- |
| 关卡一 | AP、王奶奶生命状态、恐慌值、火场风险、回合 |
| 关卡二 | AP/指挥点、老人生命稳定度、救护车倒计时、AED进度、回合 |

## 八、行动卡规范

行动卡是主要决策入口。

| 项目 | 规范 |
| --- | --- |
| 选项数量 | 每次最多4个 |
| 按钮尺寸 | 高度不低于48px |
| 图标 | 每个按钮尽量配一个明确图标 |
| 正确选项 | 不直接标绿，除非是新手教学强引导 |
| 推荐态 | 可用暖黄描边表示“花花建议方向” |
| 错误选项 | 点击后保留短暂红橙反馈，并弹出原因 |

### 1. 行动卡类型

| 类型 | 示例 |
| --- | --- |
| 判断类 | 判断现场、检查意识、观察呼吸 |
| 移动类 | 移动、低姿前进、绕行 |
| 交互类 | 拾取面罩、关闭阀门、使用AED |
| 沟通类 | 安抚沟通、说明身份、指派路人 |
| 现场管理类 | 疏散群众、维持通道、请求协助 |

## 九、AI助手“花花”规范

花花是公益提示与情绪陪伴角色，不是自动通关工具。

### 1. 提示层级

| 层级 | 触发 | 说明 |
| --- | --- | --- |
| 阶段提示 | 进入新阶段 | 告诉玩家当前目标 |
| 轻提示 | 玩家停顿或首次遇到机制 | 给方向，不给完整答案 |
| 纠错提示 | 玩家选择错误 | 解释为什么不建议这样做 |
| 风险提示 | 状态接近失败 | 提醒优先级变化 |
| 复盘提示 | 通关或失败 | 总结关键知识 |

### 2. 语气规范

- 亲切、坚定、简短。
- 不嘲讽玩家，不制造羞耻感。
- 强调“先保证安全”“按正确流程来”。
- 对危险行为直接制止，但解释原因。

## 十、QTE与特殊玩法规范

QTE用于增强参与感，不用于制造高压惩罚。

| 项目 | 规范 |
| --- | --- |
| 判定方式 | 分档评分，避免一次失败直接失败 |
| 反馈 | 用节奏光圈、音效、轻震动表达 |
| 失败处理 | 给出调整建议，允许继续 |
| 公益边界 | QTE不能取代真实知识判断 |

### 1. 关卡适配

| 关卡 | 特殊玩法 | 规范 |
| --- | --- | --- |
| 关卡一 | Fog of War、护送撤离 | 随机移动要有限制和预警 |
| 关卡二 | CPR QTE、AED步骤卡 | 玩家不能为了处理其他事中断CPR |

## 十一、错误反馈与复盘规范

错误反馈应服务教学，而不是单纯惩罚。

| 错误等级 | 表现 | 后果 |
| --- | --- | --- |
| 轻微错误 | 黄橙提示 | 扣少量评分或浪费资源 |
| 中等错误 | 红橙边框 + 花花解释 | 造成状态下降 |
| 严重错误 | 阻止执行或进入失败复盘 | 用于明显违反安全原则的行为 |

### 1. 反馈内容模板

```text
这一步不建议这样做。
原因：……
现在可以改为：……
```

### 2. 复盘必须包含

- 玩家做对的关键行为。
- 玩家可以改进的节点。
- 对应公益知识点。
- 下一次推荐流程。

## 十二、评分与结算规范

满分统一为100分，各关卡按机制拆分权重。

| 通用评分维度 | 说明 |
| --- | --- |
| 公益流程 | 是否按正确安全或急救流程行动 |
| 救援结果 | 救助对象是否稳定或撤离 |
| 风险管理 | 是否合理处理危险、障碍、群众干扰 |
| 操作表现 | QTE、路线、资源使用是否稳定 |
| 行动效率 | 是否在合理回合内完成 |

### 1. 结算页结构

| 区域 | 内容 |
| --- | --- |
| 结果标题 | 成功救援/需要复盘 |
| 核心分数 | 总分和小红花数量 |
| 关键行为 | 3-5条行为记录 |
| 知识卡 | 本关公益知识总结 |
| 重试/下一关 | 明确按钮 |

## 十三、关卡可变配置表

| 配置项 | 关卡一：烟雾厨房 | 关卡二：广场黄金四分钟 |
| --- | --- | --- |
| 地图尺寸 | 10x8 | 10x7 |
| 场景主色 | 暖灰厨房 + 烟雾灰 + 火焰橙 | 清爽广场灰蓝 + 急救红 + 安全绿 |
| 核心资源 | AP、火场风险 | AP、指挥点、救护车倒计时 |
| 救助对象状态 | 王奶奶生命状态、恐慌值、信任值 | 老人生命稳定度 |
| 特殊机制 | Fog of War、烟雾扩散、安抚、护送 | CPR QTE、AED跑腿、群众疏散 |
| 公益重点 | 火灾避险、老人沟通、安全撤离 | 心脏骤停识别、呼救、CPR、AED |
| 高分来源 | 安全判断、合理防护、成功安抚、快速撤离 | 正确流程、稳定CPR、及时AED、现场协作 |

## 十四、开发落地建议

### 1. 配置化字段

建议所有关卡使用统一配置结构：

```ts
type LevelConfig = {
  id: string
  title: string
  grid: { width: number; height: number }
  statusBar: StatusItem[]
  palette: LevelPalette
  actors: ActorConfig[]
  interactables: InteractableConfig[]
  hazards: HazardConfig[]
  actionCards: ActionCardGroup[]
  scoring: ScoreRule[]
}
```

### 2. 视觉资源命名

| 类型 | 命名示例 |
| --- | --- |
| 角色 | `actor_player_rescuer`, `actor_grandma`, `actor_bystander_a` |
| 道具 | `item_smoke_mask`, `item_aed`, `item_gas_valve` |
| 地格 | `tile_safe`, `tile_smoke`, `tile_fire`, `tile_unknown` |
| UI | `ui_action_card`, `ui_status_bar`, `ui_huahua_tip` |

### 3. 优先开发顺序

1. 搭建统一布局和状态栏。
2. 实现网格地图、格子状态和路径预览。
3. 实现行动卡系统。
4. 接入花花提示和错误反馈。
5. 实现关卡一特殊机制。
6. 用同一框架扩展关卡二特殊机制。

## 十五、验收清单

每个关卡上线前检查：

- 是否使用统一布局。
- 是否使用统一色板和状态颜色。
- 是否有清晰的顶部状态栏。
- 是否所有行动都通过行动卡或明确交互触发。
- 是否错误反馈说明了原因。
- 是否没有奖励危险或错误公益行为。
- 是否有通关/失败复盘。
- 是否能在不看外部说明的情况下理解当前目标。
